## 规则等级

- **🔴 Error**：必须修复，否则驳回/补正
- **🟡 Warning**：建议修复，有补正风险
- **🔵 Info**：参考建议

## 第一章：源代码文档（程序鉴别材料）

| 规则 | 规范值 | 来源 | 等级 |
|---|---|---|---|
| 纸张 | A4 (297mm × 210mm) | 《登记办法》第十七条 | 🔴 |
| 页边距 | 上2.5cm 下2.5cm 左2.5cm 右2.5cm（或 上2.54cm 下2.54cm 左3.17cm 右3.17cm） | 行业通用标准 | 🟡 |
| 字体 | 宋体 (Songti/SimSun) 或 等宽字体 (Courier New/Consolas) | 版权中心指南 + 行业实践 | 🟡 |
| 字号 | 五号 (10.5pt) 或 小五号 (9pt) | 行业通用标准 | 🟡 |
| 行间距 | 单倍行距 (1.0) | 行业通用标准 | 🟡 |
| 段间距 | 0磅（段前段后均为0） | 行业通用标准 | 🟡 |
| 颜色 | 黑色（禁止彩色） | 版权中心要求 | 🔴 |
| 编码 | UTF-8（禁止GBK/GB2312等可能乱码的编码） | 技术规范 | 🔴 |
| 文件大小 | 单份 ≤ 50MB | 2026年版权中心系统限制 | 🟡 |
| 加密/水印 | 禁止任何加密、第三方水印 | 2026年版权中心系统限制 | 🔴 |

### 1.2 页眉页脚

| 规则 | 规范值 | 来源 | 等级 |
|---|---|---|---|
| 页眉内容 | 软件全称 + 版本号（如"山野集-菌迹软件 V1.0"） | 版权中心 + 各地软协 | 🔴 |
| 页眉禁止 | 公司名称、URL、联系方式、"源代码"字样 | 综合实践 | 🔴 |
| 页眉对齐 | 左对齐（推荐）或居中 | 各地软协规范稍有差异 | 🟡 |
| 页码位置 | 页眉右侧（右上角）或 页脚居中 | SSIA要求右上角，部分要求页脚 | 🔴 |
| 页码格式 | "第 X 页" 或 纯数字"X" | 各地要求不同，保持一致即可 | 🔴 |
| 页脚内容 | 著作权人全称（如适用） | SSIA要求 | 🟡 |
| 页眉底线 | 页眉下方有分隔线 | 排版规范 | 🟡 |

### 1.3 内容要求

| 规则 | 规范值 | 来源 | 等级 |
|---|---|---|---|
| 总页数 | 正好 60 页（前 30 + 后 30），不足 60 页全部提交 | 《登记办法》第十条 | 🔴 |
| 每页行数 | ≥ 50 行有效代码（去除空行和注释） | 《登记办法》第十条 | 🔴 |
| 最后一页 | 最后一行必须是完整的代码模块结束（如 `};` `}` `);`） | 版权中心 + 各地软协 | 🔴 |
| 最后页行数 | ≥ 15 行（最后一页可不足50行但必须完整） | SSIA要求 | 🟡 |
| 空行 | 全部删除（允许连续代码块间最多1个空行） | 综合实践 | 🔴 |
| 注释 | 全部删除（单行// # --、多行/* */ <!-- -->，含文件头版权声明） | 版权中心要求 | 🔴 |
| 代码连续性 | 代码必须连续，不得跳跃或截断逻辑 | 《登记办法》精神 | 🔴 |
| 代码真实性 | 必须来自项目真实源文件，禁止 AI 编造 | 2026新政实质性审查 | 🔴 |
| 代码功能对应 | 源代码实现的功能必须在操作手册中有对应描述 | 补正常见原因 #08 | 🔴 |
| 代码去重 | 不能与已申请软著的代码重复（系统会查重比对） | 补正常见原因 #07 | 🟡 |
| 遗留代码排除 | 排除 mockData、模板默认文件、编译产物、node_modules | okooo5km + Fokkyp | 🟡 |
| 代码缩进 | 保持正常代码缩进，禁止大量空格或空白 | 补正常见原因 #10 | 🟡 |
| 版权声明 | 如出现 author/copyright/version 等关键字，必须与申请表一致 | SSIA要求 | 🔴 |
| 时间日期 | 代码中出现的日期必须在公司成立之后、开发完成之前 | SSIA要求 | 🔴 |
| 第三方名称 | 代码中不能出现除著作权人以外的公司或个人名称 | SSIA要求 | 🔴 |
| 开源协议声明 | 如有开源代码，必须附开源组件清单和协议说明 | 2026新政 | 🟡 |

### 1.4 代码抽取优先级

```
第一优先：入口文件（main.js, App.vue, index.ts 等）
第二优先：页面/路由文件
第三优先：核心业务逻辑组件
第四优先：API 接口封装
第五优先：状态管理
第六优先：工具函数
```

每个文件添加注释标记：`// ========== path/to/file.js ==========`（该标记不计入代码行数）


## 第四章：跨文档一致性

| 检查项 | 必须一致的位置 | 等级 |
|---|---|---|
| 软件全称 | 申请表 × 源代码页眉 × 操作手册页眉/正文/标题 | 🔴 |
| 版本号 | 申请表 × 源代码页眉 × 操作手册页眉 × 项目配置文件 × 截图中显示 | 🔴 |
| 著作权人 | 申请表 × 源代码（如出现）× 操作手册 | 🔴 |
| 源程序量 | 申请表数字 = 源代码文档首页标注行数 | 🔴 |
| 编程语言 | 申请表所选语言 = 源代码实际使用语言 | 🟡 |
| 主要功能 | 申请表描述的功能必须在操作手册中有对应章节 | 🔴 |
| 运行平台 | 申请表 × 操作手册系统要求 | 🟡 |
| 开发完成日期 | 申请表 × 操作手册中引用的日期 | 🔴 |

## 第六章：常见补正原因及预防

| 补正原因 | 占比 | 预防措施 |
|---|---|---|
| 源代码页眉格式错误（缺名称/版本号/公司名） | 35% | 验证器R-SC-03检查 |
| 截图不清晰/不匹配 | 25% | 验证器R-MA-03检查 |
| 有效代码行数不足 | 20% | 验证器R-SC-02/R-SC-08检查 |
