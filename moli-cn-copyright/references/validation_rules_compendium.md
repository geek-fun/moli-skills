| 来源 | 贡献 |

## 规则等级说明

- **🔴 Error**：必须修复，不修改会被驳回/补正
- **🟡 Warning**：建议修复，有补正风险
- **🔵 Info**：参考建议

## 一、源代码文档（程序鉴别材料）

### 🔴 R-SC-01: 文件存在性
源代码 PDF 文件必须存在，文件名包含"源代码"字样。

### 🔴 R-SC-02: 总页数 = 60 页
PDF 必须正好 60 页（前 30 页 + 后 30 页）。唯一例外：项目源代码总量本就不足 60 页时全部提供。

**来源**: okooo5km（60页硬约束）、Fokkyp、版权中心官网

**验证方法**:
```bash
mdls -name kMDItemNumberOfPages 源代码.pdf     # macOS
pdfinfo 源代码.pdf | grep Pages                # Linux
```

### 🔴 R-SC-03: 页眉规则
- 页眉必须包含 **软件全称 + 版本号**
- 格式推荐：左侧"软件全称 V1.0"，右侧"第 X 页"
- **禁止**出现公司名称、URL、无关文字
- **禁止**页眉写"源代码"字样（页眉仅标注名称和版本号）

**来源**: Fokkyp、okooo5km、软著通、行业规范

### 🔴 R-SC-04: 无空行
删除所有不必要的空行。代码块之间的自然间隔不超过 1 行。

### 🔴 R-SC-05: 无注释
删除所有注释行：
- 单行注释：`//`、`#`、`--`
- 多行注释：`/* */`、`<!-- -->`
- 文件头部的版权声明

### 🔴 R-SC-06: 以完整模块结尾
最后一页必须以代码模块的自然结尾结束（如 `}`、`);`、`end` 等）。
**禁止**截断在语句中间。

### 🟡 R-SC-07: 字体可读
建议使用宋体（SimSun/Songti）或等宽字体（Courier New/Consolas）。
字号 ≥ 小五号（9pt），推荐五号（10.5pt）。

**来源**: 版权中心官网、行业规范

### 🟡 R-SC-08: 每页有效代码 ≥ 50行
每页至少 50 行有效代码（去除空行和注释后计算）。不足则全部提交。

### 🔵 R-SC-09: 代码来源真实
代码必须来自项目源文件，禁止 AI 编造、拼凑或抄袭。
优先抽取：入口、路由、页面、核心组件、接口封装、状态管理。

**来源**: Fokkyp（代码真实性要求）

### 🔵 R-SC-10: 排除遗留代码
以下文件不应进入鉴别材料：
- `mockData*` / `mock*` / `fakeData*` — 假数据
- `utils/util.js` — 默认模板空壳
- `pages/logs/*` — 默认模板日志页
- `miniprogram_npm/` — 编译产物
- `node_modules/`、`dist/`、`build/`、`.next/`、`.nuxt/`

**来源**: okooo5km（legacy_candidates）、Fokkyp

## 二、用户操作手册（文档鉴别材料）

### 🔴 R-MA-01: 文件存在性
操作手册 DOCX（或 PDF）文件必须存在。

### 🔴 R-MA-02: 章节结构规范
必须包含以下基础章节（使用中文大写序号）：

```
一、相关文档（表格形式）
二、说明（产品定位、目标用户、业务场景）
三、功能特点（段落式，4-8 项）
四、系统要求（表格：最低/推荐配置）
五～N、具体页面/功能操作（逐章说明）
常见问题解答（3-5 个）
术语表（表格）
```

**禁止**使用阿拉伯数字编号（1. 2. 3.），应用中文大写（一、二、三、）。

**来源**: Fokkyp（manual_structure.md）、okooo5km

### 🔴 R-MA-03: 截图要求
每个核心功能页面必须有截图。如果截图尚未准备，必须有可见的截图预留文字：
```
【截图预留：请在空白处插入XXX截图。】
```
**禁止**使用 HTML 注释 `<!-- -->` 作为截图占位。

### 🟡 R-MA-04: 页数 ≥ 15 页
操作手册总页数建议不少于 15 页（截图充实后通常可达 20-30 页）。

### 🟡 R-MA-05: 段落式写作
正文以自然段落为主，避免大量使用：
- 项目符号列表（`-`、`*`、`•`）
- 编号步骤列表（`1. 2. 3.`）
- 字段模板（"进入方式："、"操作步骤："）

### 🟡 R-MA-06: 去 AI 味
避免以下套话：
`旨在`、`赋能`、`一站式`、`高效便捷`、`显著提升`、`强大能力`、`丰富功能`、`智能化`

### 🔵 R-MA-07: 写作口径
- 每个核心页面覆盖：使用场景 → 页面用途 → 进入位置 → 用户动作 → 系统反馈
- 语言面向普通用户，**避免**技术术语（代码、框架、接口、状态管理、异步）
- 功能描述基于**项目真实源码**，不编造不存在功能

**来源**: Fokkyp（business_understanding_rules.md）、okooo5km

## 三、申请表信息

### 🔴 R-AP-01: 软件全称以"软件"结尾
正确示例：`山野集-菌迹软件`、`智慧记账软件`
**禁止**：`山野集-菌迹`、`山野集-菌迹APP`

**来源**: okooo5km、版权中心官网

### 🔴 R-AP-02: 版本号格式 V1.0
- 格式：`V` + 主版本 + `.` + 次版本
- 首次申请一律填 V1.0（如果项目版本号小于 V1.0 需询问用户）
- **禁止**三位格式（V1.0.0）

### 🔴 R-AP-03: 主要功能 500-1300字
主要功能描述必须 ≥ 500 字且 ≤ 1300 字。建议落在 500-600 字。

**来源**: okooo5km（原200字→调整为500-1000字）、2026新政

### 🟡 R-AP-04: 源程序量填写正确
- `源程序量` = 项目全部源程序总行数（非鉴别材料抽取行数）
- 申请表数字 = 源程序文档首页标注的数字

### 🟡 R-AP-05: 日期逻辑正确
`开发完成日期 ≤ 首次发表日期 < 申请日期`

**来源**: okooo5km

### 🔵 R-AP-06: 软硬件环境 ≤ 50字符
每个环境字段不超过 50 个字符。

### 🔵 R-AP-07: 编程语言选择
从预设列表选择 + 自定义 ≤ 120 字符。

## 四、跨文档一致性

### 🔴 R-CO-01: 软件名称一致
以下位置必须使用完全相同的软件全称：
- 申请表「软件全称」字段
- 源代码 PDF 页眉
- 操作手册页眉、标题、正文

### 🔴 R-CO-02: 版本号一致
以下位置的版本号必须完全一致（含 V 的大小写）：
- 申请表「版本号」字段
- 源代码 PDF 页眉和首页
- 操作手册页眉
- 项目配置文件（`package.json`、`config.js` 等）
- 运行截图中的版本号

**来源**: okooo5km（版本号一致性核验）、Fokkyp

### 🟡 R-CO-03: 著作权人一致
所有文档首页的著作权人名称必须一致。

### 🟡 R-CO-04: 源程序量一致
申请表中的源程序量 = 源程序文档首页标注的行数。

## 五、2026 年新政

### 🟡 R-PL-01: AI 辅助开发声明
如果软件使用了 AI 辅助开发（非 AI 全量生成），必须提交《AI辅助开发声明》：
- 明确 AI 使用范围和辅助内容
- 人工研发占比 ≥ 70%
- 无声明则按 AI 全量生成判定

**来源**: 2026年3月15日版权中心新政

### 🟡 R-PL-02: 开源组件说明
如果软件使用开源组件，必须列明：
- 组件名称
- 版本号
- 来源
- 许可协议

未标注视为抄袭。

### 🟡 R-PL-03: 开发周期合理
开发完成日期与复杂度匹配：
- 简单工具软件：建议 ≥ 15 天
- 常规办公软件：建议 ≥ 30 天
- 复杂系统（ERP/CRM）：建议 ≥ 60 天

### 🔵 R-PL-04: 补正次数限制
补正次数不超过 2 次，超过直接驳回。提交前务必按本清单逐项自查。

### 🔵 R-PL-05: 签章要求
2026年3月15日起使用新版申请表，签章页需：
- 经办人亲笔手写承诺
- 签名并填写身份证号
- 缺一不可

## 六、附录：常见驳回原因统计

| 原因 | 占比 | 说明 |

| 源代码页眉格式错误 | 35% | 缺软件名称、版本号，或格式不规范 |
| 说明书截图不清晰 | 25% | 截图模糊、UI与实际版本不匹配 |

## 七、验证命令

```bash
# 一键验证当前目录下的所有软著材料
python3 ${MOLI_SKILLS_DIR}/moli-cn-copyright/scripts/validate_materials.py \
  --software-name "山野集-菌迹软件" \
  --version V1.0

# 指定材料目录
python3 ${MOLI_SKILLS_DIR}/moli-cn-copyright/scripts/validate_materials.py \
  --workdir /path/to/materials \
  --software-name "山野集-菌迹软件" \
  --version V1.0

# JSON 格式输出
python3 validate_materials.py \
  --software-name "山野集-菌迹软件" \
  --version V1.0 \
  --format json \
  --output report.json

# 仅检查源代码
python3 validate_materials.py \
  --software-name "山野集-菌迹软件" \
  --version V1.0 \
  --mode source-only
```
